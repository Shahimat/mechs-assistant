# Задача для саморазвития: спроектировать реляционную БД кланового склада

**Для кого:** тот, кто учится проектировать реляционные БД. Начинающий
backend-разработчик, любопытный фронтендер, аналитик — кто угодно.
**Что делаем:** тренируем мышление на реальном домене — схема таблиц,
связи, индексы, запросы.
**Как относиться:** это не проверочная работа — это песочница. Ошибки
и переделки — часть процесса. Готового «правильного ответа» нет, есть
разные компромиссы; главное — уметь объяснить свой выбор себе самому.

## 0. Инструменты и локальная БД

### Какую СУБД взять для учёбы

**SQLite** — в файле, ставить ничего не надо (уже есть в macOS/Linux
из коробки; на Windows — один exe). Идеально для учебы:

- Zero setup: `sqlite3 warehouse.db` в терминале — файл создаётся сам.
- Совпадает с продом. В нашем проекте бэкенд будет на **Cloudflare
  D1** (SQLite-совместимая, но serverless). То, что напишешь
  локально, ляжет в prod без переделок.
- Ограничения SQLite полезно словить сразу, не в проде:
  - нет ENUM → `TEXT` + `CHECK (col IN ('a','b','c'))`;
  - нет ARRAY → линковочные таблицы (это и правильно с точки зрения
    нормализации);
  - нет BOOLEAN → `INTEGER` 0/1;
  - даты хранят как `TEXT` в ISO-формате (`'2026-02-05'`).

**Postgres в Docker** — избыточно для учёбы схемы. Возьмёшь его,
когда фичи потребуют (row-level security, JSON-типы, arrays). Пока
не нужно.

### DBeaver — рабочий GUI

Скачай [DBeaver Community](https://dbeaver.io/download/) (бесплатно,
все ОС). Одно приложение = все СУБД.

Подключение к SQLite за 30 секунд:

1. **File → New → DBeaver → Database Connection**.
2. Тип: **SQLite**.
3. **Create** → указать путь к будущему `warehouse.db` (пустой файл
   тоже подойдёт — SQLite создаст его при первом подключении).
4. **Finish**.

Что делать в DBeaver:

- **SQL Editor** (Ctrl+Enter — запустить выделенный запрос) — для
  всех `CREATE TABLE`, INSERT, SELECT.
- **ER Diagram** (правый клик на схеме → View Diagram) — визуальная
  проверка связей после `CREATE TABLE`. Полезно для отладки: FK, не
  замеченные глазами, здесь видны стрелочками.
- **Data Editor** — редактировать строки как в Excel, тестировать
  вставку тестовых данных.

Полезно поставить [расширение
Mermaid](https://dbeaver.com/docs/dbeaver/Working-with-diagrams/) для
экспорта диаграммы в текст.

### Как воркать над задачей

Задача разбита на 4 шага. После каждого — короткая самопроверка;
только когда сходится — двигайся дальше. Возвращаться к предыдущему
шагу и переделывать — норма.

```
Шаг 1 — ER-диаграмма       (концептуально: сущности + связи)
   ↓ самопроверка
Шаг 2 — Нормализация       (проверить 1NF, 2NF, 3NF)
   ↓ самопроверка
Шаг 3 — SQLite-схема       (CREATE TABLE с типами, FK, soft-delete)
   ↓ самопроверка
Шаг 4 — Тест-кейсы         (INSERT + SELECT, сверить с эталоном)
```

Всё это — в одном `warehouse.db` файле. Легко скопировать, поделиться
с кем-то для обсуждения или отложить и вернуться завтра.

## 1. Домен за 30 секунд

Клан игры «Мехи.Земля» (15-30 человек) ходит на рейды. С рейдов
приносят лут. Часть лута штучная — «1 Хребет Ройтука на 4 игрока» —
сразу не поделить. Нужна книга учёта: кто сколько **внёс** и кто
сколько **взял**, чтобы делить со временем справедливо.

### Жизненный цикл одной единицы лута

```
   ┌─────────────────┐
   │  Приход после   │  Пример: 1 Хребет Ройтука,
   │     рейда       │           4 участника
   └────────┬────────┘
            │
            │ каждому участнику начисляется 1/4 = 0,25 доли
            ▼
   ┌─────────────────┐
   │   Баланс копится│  Игрок А: 0,25 → следующий рейд → 0,50 → ... → 1,0
   │  за много рейдов│
   └────────┬────────┘
            │
            │ баланс ≥ 1 → игрок может забрать
            ▼
   ┌─────────────────┐
   │     Выдача      │  Игрок А списывает 1 хребет со склада,
   │    со склада    │  с его баланса вычитается 1
   └─────────────────┘
```

### Кто с чем связан (концептуально)

```
   Участник                            Предмет каталога
       │                                    │
       │ участвует в                        │ используется в
       ▼                                    ▼
   Приход  ◄──── содержит N предметов ────►
       │                                    │
       ▼                                    │
   Выдача (кто-то из участников) ───────────┘
```

Три ключевые сущности: **участники клана**, **приходы** (со списком
участников), **выдачи** (кто забрал). Предметы — из внешнего каталога.

## 2. Пример данных

**Участники** (у нас 3 для примера):

| id  | ник   | роль   |
| --- | ----- | ------ |
| 1   | user1 | admin  |
| 2   | user2 | editor |
| 3   | user3 | viewer |

**Приходы:**

| дата       | предмет        | кол-во | участники           |
| ---------- | -------------- | ------ | ------------------- |
| 2026-01-10 | Хребет Ройтука | 1      | user1, user2, user3 |
| 2026-01-15 | Хребет Ройтука | 2      | user1, user2        |
| 2026-02-01 | Хребет Ройтука | 1      | user1               |

**Выдачи:**

| дата       | предмет        | кол-во | кто   |
| ---------- | -------------- | ------ | ----- |
| 2026-02-05 | Хребет Ройтука | 1      | user1 |

### Расчёт баланса user1'а по «Хребет Ройтука»

```
приход 10.01: 1 шт × (1/3) = 0,333
приход 15.01: 2 шт × (1/2) = 1,000
приход 01.02: 1 шт × (1/1) = 1,000
                             ------
                    начислено: 2,333
выдача 05.02:                −1,000
                             ------
          доступно сейчас:    1,333   ← ≥ 1, ещё может взять один
```

«На складе всего»: `Σ приход − Σ выдача = (1+2+1) − 1 = 3`.

> Когда напишешь свои `CREATE TABLE` — залей эти же данные
> INSERT'ами в DBeaver и убедись, что твой SELECT-баланс даёт
> `earned = 2.333`, `taken = 1`, `available = 1.333`. Если
> получилось — схема живёт.

## 3. Требования

**Функциональные:**

1. Записать приход: `date, item, qty, participants[]`.
2. Записать выдачу: `date, item, qty, member`.
3. Отчёт «сколько на складе».
4. Отчёт «мой баланс» по каждому предмету.
5. История операций участника.
6. Пометить старую запись как ошибочную (без физического удаления).
7. Управление участниками: добавить, отключить, сменить роль.

**Нефункциональные:**

- **Целостность.** Нельзя удалить участника с историей — только
  «отключить».
- **Аудит.** У каждой строки: кто создал/когда, кто удалил/когда.
- **Каталог предметов** уже существует вне БД (публичные JSON'ы). БД
  ссылается по `item_key: string` (translit имени, напр.
  `khrebet_roytuka`). Свою таблицу `items` **не** заводим.

> Явные индексы и транзакции сознательно вне этой задачи. На нашем
> масштабе (15-30 юзеров, ~25k строк за 5 лет) SQLite прекрасно
> справится без ручных `CREATE INDEX`, а транзакции в проде Drizzle
> оборачивает сам через `db.transaction()`. Тема углубления —
> см. раздел 10.

## 5. Шаг 1. ER-диаграмма

**Цель шага:** описать домен на концептуальном уровне —
_какие сущности есть_ и _как они связаны_, без типов и деталей
реализации.

### Что такое ER-диаграмма (двумя предложениями)

Сущности — прямоугольники (`member`, `income_event`, ...). Связи —
линии между ними, с указанием мощности (1:1, 1:N, N:M).

Пример нотации:

```
   member  ──1─────N──►  income_event.created_by
           ──1─────N──►  withdraw_event.created_by
           ──N─────N──►  income_event  (через линковочную таблицу)
```

### Задание

1. Выпиши **все сущности**, которые нужны для функциональности из
   раздела 3. Не пропускай линковочную таблицу — связь «многие ко многим» = это
   отдельная сущность.
2. Для каждой сущности выпиши **ключевые атрибуты** (без типов).
   Пример: `member { id, nickname, role, is_active }`.
3. Для каждой пары связанных сущностей укажи **мощность** (1:1, 1:N,
   N:M) и направление.
4. Нарисуй — на бумаге, в draw.io, в [dbdiagram.io](https://dbdiagram.io/),
   в DBeaver (View → New ER Diagram). Формат не важен.

### Самопроверка (шаг 1)

- [ ] Есть сущность «участники клана».
- [ ] Есть отдельная сущность «приход» (одна запись рейда).
- [ ] Есть отдельная сущность «выдача».
- [ ] Есть линковочная таблица для связи «приход ↔ участники» (N:M).
- [ ] Есть связь «кто создал запись» (для аудита).
- [ ] Предметы — **не** отдельная сущность в БД (внешняя ссылка по
      key, помечена как external).
- [ ] Для каждой связи проставлена мощность.

Если все пункты закрыты — переходим к нормализации.

## 6. Шаг 2. Нормализация — проверить 1NF, 2NF, 3NF

**Цель шага:** проверить, что ER не содержит скрытых проблем
(массивов в ячейке, повторяющихся групп, транзитивных зависимостей).

Каждая форма — вопрос к схеме.

### 1NF (первая нормальная форма)

**Вопрос:** каждая ячейка содержит одно атомарное значение?

Плохо:

```
income_event { id, item_key, qty, participants: "1, 2, 3" }
```

Причина плохо: чтобы посчитать баланс — придётся парсить строку
внутри SQL. Индексы не работают, целостности нет.

Хорошо: отдельная линковочная таблица `income_participants`.

### 2NF (вторая нормальная форма)

**Вопрос:** каждый non-key атрибут зависит от **всего** первичного
ключа, а не от его части?

Актуально только для составных ключей. Пример:

```
income_participants { income_event_id, member_id, member_nickname }
                                                  ^^^^^^^^^^^^^^^
                       зависит только от member_id, не от income_event_id
                       → должно жить в members, не здесь
```

### 3NF (третья нормальная форма)

**Вопрос:** нет ли транзитивных зависимостей (когда атрибут зависит
от non-key атрибута)?

Плохо:

```
income_event { id, created_by_email, created_by_nickname, created_by_role }
                                    ^^^^^^^^^^^^^^^^^^  ^^^^^^^^^^^^^^^
                    nickname и role зависят от email (косвенно, через
                    member), а не от id → жить должны в members
```

Хорошо: `income_event { id, created_by_id → members.id }`. Никнейм и
роль достаём JOIN'ом.

### Задание

Пройди по каждой сущности из шага 1. Задай ей три вопроса выше.
Если находишь нарушение — исправь ER и вернись.

### Самопроверка (шаг 2)

- [ ] Ни в одной сущности нет multi-value атрибутов
      («через запятую», JSON-массивов).
- [ ] У сущностей с составным PK каждый non-key атрибут зависит от
      **всего** ключа.
- [ ] Нет продублированных данных, вытекающих из FK
      (никакого `created_by_nickname` рядом с `created_by_id`).

Если все три закрыты — можно писать SQL.

## 7. Шаг 3. SQLite-схема (`CREATE TABLE`)

**Цель шага:** превратить нормализованную ER в физическую схему для
SQLite. Здесь появляются типы, FK-стратегии, soft-delete, аудит-поля.

### Развилки, которые предстоит разрешить

Это не единственно правильные ответы — выбирай осознанно.

**А. Приход и выдача — одна таблица или две?**

```
Вариант 1:                     Вариант 2:
─────────────────              ─────────────────
events                         income_events
  kind: 'in' | 'out'           withdraw_events
  qty: signed                    (две отдельные)

+ проще запрос «баланс»        + чёткие типы полей
- полиморфизм по kind            (у прихода participants,
- разная структура связей         у выдачи только member)
  всё равно нужна              - Σ баланса — UNION/JOIN
```

**Б. Долю (К-т) хранить или считать?**

`share = 1 / count(participants)`.

- Считать на лету — просто, но при 5k событий тяжелее.
- Хранить в линковочной таблице (`income_participants.share`) — быстрее read,
  сложнее write (пересчитывать при добавлении/удалении участника).

**В. Soft-delete — как?**

Варианты: `is_deleted BOOL` / `deleted_at TIMESTAMP` / отдельная
таблица. Какой лучше и почему?

**Г. Как хранить типы, которых нет в SQLite?**

- ENUM (role, status) → `TEXT` + `CHECK (col IN ('a','b','c'))`.
- BOOL → `INTEGER` 0/1.
- Даты → `TEXT` в ISO-формате `'YYYY-MM-DD'` или `'YYYY-MM-DD
HH:MM:SS'`.

**Д. Для каждого FK — какой `ON DELETE`?**

- `RESTRICT` — блокирует удаление родителя, если есть ссылки.
- `SET NULL` — при удалении родителя ссылка становится NULL.
- `CASCADE` — при удалении родителя удаляются и «дети».

Обычно для аудит-таблиц `RESTRICT` (никогда не терять историю), для
линковочных таблиц — `CASCADE` (нет смысла хранить связку без родителя).

### Задание

1. Определись по каждой из развилок А-Д.
2. Напиши `CREATE TABLE` для всех сущностей: типы, `NOT NULL`,
   `DEFAULT`, `PRIMARY KEY`, `FOREIGN KEY … ON DELETE …`,
   `CHECK (…)` для enum'ов.
3. Добавь аудит-поля: `created_by`, `created_at`, `deleted_by?`,
   `deleted_at?` на таблицах, где важна история.
4. Выполни `CREATE TABLE` в DBeaver, посмотри в ER Diagram — связи
   правильные?

### Самопроверка (шаг 3)

- [ ] Ни в одной ячейке нет массивов / «через запятую» / JSON-хаков.
- [ ] Каждое отношение выражено через FK.
- [ ] Для каждого FK осознанно выбрал `ON DELETE …` и мог бы
      объяснить, почему.
- [ ] Есть аудит-поля на таблицах, где важна история.
- [ ] Soft-delete реализован там, где физическое удаление
      недопустимо.
- [ ] Enum'ы через `TEXT + CHECK`, а не сырые строки без
      ограничения.
- [ ] Каждое поле в схеме можешь объяснить одним предложением: «оно
      нужно, потому что…».

Если все закрыты — переходи к тестам.

## 8. Шаг 4. Тест-кейсы (SQL)

**Цель шага:** прогнать заранее посчитанные сценарии — убедиться,
что схема даёт правильные результаты.

**Про плейсхолдеры.** Ты сам придумал имена таблиц и колонок — я не
знаю их наперёд. В SQL ниже используется `<placeholder>` — замени на
своё:

- `<members>`, `<income>`, `<withdraw>`, `<income_participants>` —
  твои имена таблиц.
- `<nickname_col>`, `<role_col>`, `<item_col>`, `<qty_col>`,
  `<date_col>`, `<share_col>`, `<deleted_at_col>` — твои имена
  колонок.
- Если у тебя одна таблица `<events>` вместо двух — адаптируй
  соответственно.

Порядок тест-кейсов: 1 → 2 → 3 → 4. Каждый предыдущий — предпосылка
следующего (данные не сбрасываются).

### Тест-кейс 1 — базовый баланс

**Setup** (INSERT'ы):

```sql
-- Участники
INSERT INTO <members> (id, <nickname_col>, <role_col>) VALUES
  (1, 'user1', 'admin'),
  (2, 'user2', 'editor'),
  (3, 'user3', 'viewer');

-- Приходы
INSERT INTO <income> (id, <date_col>, <item_col>, <qty_col>) VALUES
  (101, '2026-01-10', 'khrebet_roytuka', 1),
  (102, '2026-01-15', 'khrebet_roytuka', 2),
  (103, '2026-02-01', 'khrebet_roytuka', 1);

-- Участники приходов (линковочная таблица)
INSERT INTO <income_participants> (<income_id_col>, <member_id_col>) VALUES
  (101, 1), (101, 2), (101, 3),
  (102, 1), (102, 2),
  (103, 1);

-- Выдачи
INSERT INTO <withdraw> (id, <date_col>, <item_col>, <qty_col>, <member_id_col>) VALUES
  (201, '2026-02-05', 'khrebet_roytuka', 1, 1);
```

**Check 1a — баланс user1 по «khrebet_roytuka»:**

```sql
-- Ожидаемый результат: одна строка, available ≈ 1.333
SELECT
  m.id AS member_id,
  '<item_key>' AS item,
  <вычисление earned>  AS earned,
  <вычисление taken>   AS taken,
  <вычисление earned> - <вычисление taken> AS available
FROM <members> m
WHERE m.id = 1;
```

Проверка на живой схеме — заполни выражения `<…>`. Правильные числа:
`earned = 2.333, taken = 1, available = 1.333`.

**Check 1b — «сколько на складе» по всем предметам:**

```sql
-- Ожидаемый результат: одна строка (khrebet_roytuka, 3)
SELECT <item_col>,
       SUM(<qty_col>) - COALESCE((
         SELECT SUM(w.<qty_col>)
           FROM <withdraw> w
          WHERE w.<item_col> = i.<item_col>
       ), 0) AS on_stock
  FROM <income> i
 GROUP BY <item_col>;
```

Правильное число: `khrebet_roytuka = 3`.


**Check 2 — баланс всех участников по «chasti_pirata_sat»:**

```sql
-- Ожидается 3 строки, значения available: user1=2.667, user2=0.667, user3=0.667
SELECT
  m.id, m.<nickname_col>,
  <вычисление earned> AS earned,
  <вычисление taken>  AS taken,
  <вычисление earned> - <вычисление taken> AS available

WHERE m.id IN (1, 2, 3)
ORDER BY m.id;
```

### Тест-кейс 3 — soft-delete

**Setup** — отменяем приход №102 (ошибочный):

```sql
UPDATE <income>
   SET <deleted_at_col> = '2026-02-10',
       <deleted_by_col> = 1        -- user1 (admin) пометил
 WHERE id = 102;
```

**Check 3 — тот же SELECT «баланс всех по khrebet_roytuka»:**

```sql
-- Ожидается: available user1=0.333, user2=0.333, user3=0.333

  <вычисление earned WHERE deleted_at IS NULL> AS earned,
  <вычисление taken WHERE deleted_at IS NULL>  AS taken,

FROM <members> m
WHERE m.id IN (1, 2, 3);
```

Ключевой момент: в подзапросах должно быть

**Check 3b — на складе после отмены:**

```sql

SELECT <item_col>, SUM(<qty_col>) - COALESCE((
         SELECT SUM(w.<qty_col>) FROM <withdraw> w
          WHERE w.<item_col> = i.<item_col> AND w.<deleted_at_col> IS NULL
       ), 0) AS on_stock
  FROM <income> i
 WHERE i.<item_col> = 'khrebet_roytuka' AND i.<deleted_at_col> IS NULL
```

## 10. Что читать, если что-то из этого незнакомо

- Нормализация 1NF/2NF/3NF.
- Линковочные (many-to-many) таблицы.
- `ON DELETE RESTRICT / CASCADE / SET NULL`.
- Паттерны soft-delete.

Не гугли готовые «warehouse schema» — они про склад товаров, не про
кланы. Запутают.

## 11. Куда двигаться дальше

Если задача зашла и хочется углубиться — вот направления по
возрастающей сложности:

1. Переписать схему через **Drizzle ORM** — TypeScript-типы вместо
   голого SQL. Появится type-safety при запросах.
2. Написать **миграции** (одна `init` + следующие). Понимание, как
   схема эволюционирует со временем.
3. Написать **unit-тесты** на инварианты: «выдача больше баланса
   отклоняется», «удаление активного участника отклоняется» и т.п.
4. Обернуть в **HTTP endpoints** — REST или tRPC — и прикрутить
   авторизацию.
5. Развернуть на **Cloudflare D1** и связать с реальным фронтом.
6. **Транзакции.** Разобраться с `BEGIN / COMMIT / ROLLBACK` в
   SQLite и `db.transaction(...)` в Drizzle. Понять, зачем нужна
   атомарность и когда её отсутствие даёт битые данные.
7. **Индексы и `EXPLAIN QUERY PLAN`.** Прогнать нагрузочный тест:
   сгенерировать ~100k фиктивных записей, посмотреть какие запросы
   тормозят, добавить целевые индексы, сравнить `SCAN` vs `SEARCH` в
   плане выполнения.

Каждый шаг — самостоятельная тренировка. Спешить некуда.
