# Протоколы сценариев

Определяй сценарий по ключевым словам сообщения юзера и следуй протоколу
ниже. Не приступай к `execute`, пока не согласован план.

## Чтение перед правкой

Действует на любой сценарий с правками — включая «почини», «поправь»,
«доделай», где явного `plan`/`execute`-ключа нет.

1. Пройти порядок чтения из `CLAUDE.md` целиком, не выхватывая файлы
   «на глаз».
2. В ответе одной строкой перечислить: что прочитал по этому порядку и
   что пропустил осознанно с обоснованием («workflow, project, rspack,
   verification; conventions по MUI и TanStack не читал — правки их не
   касаются»).
3. Если в ходе работы задел новый слой (новый namespace, новая
   convention) — дочитать до следующего Edit'а, не после.

## Атомарность фаз и коммитов

Правки атомарны по типу изменения. Каждая фаза = ровно один тип по
karma commit-msg: `docs` / `fix` / `refactor` / `feat` / `test` /
`perf` / `chore` / `style`. Смешивать типы в одной фазе нельзя. Как
юзер собирает фазы в коммиты (по одному на фазу или батчем) — его
решение; агент всё равно идёт фазами и показывает отчёт после каждой.

Дефолтный порядок для сложной задачи:

1. `docs` — сначала выравниваем правду в проектной документации. Сюда же
   попадают записи в `CHANGELOG.md`, если изменение подпадает под его
   критерий (см. convention `changelog`).
2. `fix` — существующие баги, всплывшие при анализе.
3. `refactor` — снимаем инфраструктурные ограничения, готовим почву.
4. `feat` — новая функциональность. **Никакого рефакторинга внутри `feat`.**

Отклонения от порядка — только с явным обоснованием.

`CHANGELOG.md` трогают **только** в фазе `docs`. В `feat` / `fix` /
`refactor` / `test` / `perf` / `chore` / `style` — не трогают, чтобы не
смешивать типы изменений.

Каждая фаза заканчивается точкой согласования: короткий отчёт «сделал
X, готов к фазе Y» → ждать «ок» → идти дальше. Не переходить к
следующей фазе автоматически.

### Спецификация впереди реализации

Когда фаза `docs` обновила `entities/*.yml` / `behavior/*.yml` до
состояния, которого код ещё не описывает, — это норма для дефолтного
порядка. В затронутых `behavior/*.yml` — `verified: false` +
`verified_note` вида «спецификация обновлена в фазе docs, реализация
в фазе feat». По закрытии фазы реализации — `audit` по затронутым
behavior, `verified: true` + `verified_at`, снятие пометки из
`verified_note`.

Если решение подпадает под критерий `CHANGELOG.md` (крупное
архитектурное, смена стека, крупные правки `context/`, веха — см.
convention `changelog`), запись в журнал появляется в этой же фазе
`docs`, вместе с правкой доки. В последующих `feat` / `fix` /
`refactor` `CHANGELOG.md` не трогают.

### Инъекция подфазы `docs`

Если в процессе фазы N (напр. `feat`) выяснилось, что проектная
документация не покрывает случай — не «дописываем на ходу»:

1. Стоп фазу N. `docs` и текущий тип в одном заходе не смешиваем.
2. Откатить незакоммиченные изменения фазы N (нетривиальный откат
   согласовать с юзером).
3. Инъектировать подфазу `docs`: отдельное сообщение с планом правок
   доки → согласование → правки → отдельная точка согласования.
4. Вернуться к фазе N поверх обновлённой доки.

## `plan` (ключи: «предложи», «как лучше», «варианты», «что думаешь»)

1. Загрузить relevant entities / behaviors / conventions по namespace задачи
   (через `context/index.yml`, не `ls`).
2. Явно перечислить расхождения между «как есть» и «как хочу».
3. Дать 1–3 варианта решения по 2–4 строки каждый, с trade-off.
4. Одной строкой — своя рекомендация.
5. Если решение задевает больше одного типа изменений (`docs`, `fix`,
   `refactor`, `feat`, ...) — предложить разбивку на атомарные фазы
   (см. «Атомарность фаз и коммитов»). Порядок фаз — по дефолту либо с
   явным обоснованием отклонения.
6. Распределить план по целям (`context/views/goals.yaml` +
   `goal--<key>.yaml`): в какую существующую цель ложатся задачи. Нет
   подходящей — предложить создать новую цель (см. convention `goals`).
   Прогнать `goal-consistency` — нет ли противоречия с другими целями
   (конфликт интентов, дубль задачи, цикл зависимостей).
7. Ждать выбор юзера. Не приступать к execute автоматически.

## `execute` (ключи: «делай», «поехали», «внедряй»)

1. Работать только после подтверждения плана и после чтения проектной
   документации по namespace задачи (см. «Чтение перед правкой»).
2. Если задача разбита на фазы — идти строго по одной фазе за раз, не
   смешивая типы изменений (см. «Атомарность фаз и коммитов»). В конце
   каждой фазы — точка согласования, ждать «ок» перед переходом.
3. Между шагами — короткие статус-строки (одна строка на шаг).
4. Критические алёрты (что-то опасное, спорное, неожиданное) — стоп,
   вопрос юзеру одной строкой.
5. В конце — обновить структуру целей (`context/views/goals.yaml` +
   затронутые `goal--<key>.yaml`: снять закрытые задачи, при полном
   закрытии цели — CHANGELOG + удалить файл цели, см. convention
   `goals`; при правке целей прогнать `goal-consistency`), затронутые
   `docs/behavior/*.yml` (`verified`), `docs/entities/*.yml`
   (спецификация), `context/index.yml` (при создании / удалении /
   переименовании сущности), `CHANGELOG.md` (только в фазе `docs`,
   если решение подпадает под критерий журнала — см. convention
   `changelog`).

## `audit` (ключи: «проверь состояние», «где мы сейчас», «прогони проверку»)

1. Пройтись по behaviors в интересующем namespace.
2. Выполнить сценарии проверок, где возможно (`grep`, чтение кода,
   `npm test`, `npm run build`, `npm run lint`).
3. Обновить `verified` и `verified_at` по факту.
4. Отчёт юзеру: сколько проверено, где расхождения с entity, где
   `verified: false` остаётся и почему.

## `challenge` (ключи: «покритикуй», «поспорь», «найди слабое место»)

1. По умолчанию не соглашаться.
2. Минимум 2 контраргумента к предложенному подходу.
3. Проверить противоречие с conventions — если конфликт, указать, с
   какой именно convention.
4. В итоге: «ок с оговорками X» или «альтернатива Y».

## `report` (ключи: «что сейчас активно», «где мы», «статус»)

1. **Не пересказывать цели дословно.**
2. Читать оба источника задач (см. convention `todo-split`): структуру
   целей (`context/views/goals.yaml` + релевантные `goal--<key>.yaml`,
   dev/internal) и `data/overrides/todos.yml` (product-facing из Sheets).
3. Выдать 3–5 задач в формате: `→ действие → пояснение одной строкой`.
   Пометить, откуда пункт (dev или public), если это неочевидно.
4. Добавить блок «блокеры».
5. Одной строкой — «что делать первым».

## Синхронизация с источниками

Реквизиты: блок `sync` в `context/project.yml` (`index_source`,
`index_sha`, `contract.key`, `contract.synced_sha`, `synced_at`) + при
подключённых модулях — их manifests в `context/modules/<key>.yml`
(`source`, `synced_sha`, `synced_at`, `provides`).

### Fetch — только `curl`

Единственный разрешённый fetch-канал для source-sync — `curl` через
Bash с фиксированными флагами:

```bash
curl -fsSL -H "Cache-Control: no-cache" -H "Pragma: no-cache" "<url>" -o "$tmp"
shasum -a 256 "$tmp" | awk '{print $1}'
```

**WebFetch для source-sync запрещён.** Он подставляет кэш, режет и
переформатирует контент → SHA-checkpoint от такого ответа не сойдётся с
SHA-256 raw-контента источника.

### Команда «сверь всё» (ключи: «сверь спецификацию с источниками», «синхронизируй», «обнови из источника», «проверь источник»)

Единый протокол — за один заход обходит контракт и все подключённые
модули через индекс:

1. Прочитать `sync.index_source` из `context/project.yml`. Пусто →
   «источник не сохранён, пришли ссылку», дальше не идти. При первом
   получении ссылки — сохранить.
2. `curl` → тянем `index.yml` во временный файл, считаем SHA-256.
3. Совпал с `sync.index_sha` → «без изменений», выход. Ни контракт, ни
   модули не тянем.
4. Не совпал → читаем индекс. Для каждого подключённого источника
   (контракт из `sync.contract.key`; модули из `context/modules/*.yml`)
   находим актуальный `sha256` в индексе.
5. Сравниваем `sha256` из индекса с `synced_sha` в проекте:
   - совпал → источник не менялся, пропускаем;
   - разошёлся → `curl` тянет конкретный файл (`raw` из индекса), diff,
     план миграции по слоям.
6. Дополнительно по индексу:
   - подключённый модуль пропал из индекса → предупредить юзера;
   - `applicable_to` подключённого модуля больше не содержит ключ
     подключённой базы → предупредить юзера;
   - в индексе появились новые модули с подходящим `applicable_to` →
     упомянуть как справку («опционально можно подключить»), не
     подключать сам.
7. Расхождений нет → «без изменений» одной строкой.
8. Расхождения есть → единый план миграции: конкретные правки по слоям и
   файлам, какие behaviors получат `verified: false` (по правилам
   convention `verification`). Дождаться явного «да» до внесения.
9. После применения обновить все SHA: `sync.index_sha`,
   `sync.contract.synced_sha`, `synced_sha` в manifest'ах затронутых
   модулей, `synced_at` — на текущую ISO-дату.

### Проверка `key` при отдельной тяге

Если тянешь контракт или модуль отдельно (не через индекс) — обязательно
сверь поле `key` из frontmatter источника с сохранённым ключом
(`sync.contract.key` или ключ в manifest'е модуля). Не совпало — стоп,
сообщи юзеру, не применяй. Защита от подмены URL.

### Смысл механизма

Один шорткат вместо ручного пересылания ссылки при каждом обновлении.
SHA-checkpoint экономит токены: не совпал только один из источников —
читаем и анализируем только его.

## Git и изменения истории

Коммитит и пушит юзер сам. Не спрашивать «может, закоммитим?» после
работы — короткого отчёта (что сделано + при нужде `git status`)
достаточно. Подробности — в convention `git-commits`.

Если юзер сам явно просит выполнить git/arc-операцию (`commit`, `push`,
`merge`, `rebase`, `reset`, `checkout -b`, `tag`, `stash drop/pop` и
т.п.), действуем по глобальному правилу `~/.claude/CLAUDE.md`: сначала
одним сообщением показать полный список команд, запускать после явного
«да».

## Git-процесс проекта (ветки, ручки, защита main)

Реализованная модель разработки и CI (с 2026-07-15). Процедурная
настройка — `docs/manuals/setup-github-workflow.md`.

**Ветки.** Дефолтная рабочая ветка — `develop`; `main` = «то, что
задеплоено». Все правки идут в `develop`, в `main` вливаются только
через PR (merge через веб-UI).

**Защита `main`.** Ruleset на `main` с `Enforcement: Active`:
`Require a pull request before merging` (approvals = 0) + `Restrict
deletions` + `Block force pushes`. Bypass — роль `Write` со статусом
`Always allow`, под неё попадают автор-admin (Write+ автоматически) и
`GITHUB_TOKEN` sync-workflow'ов. Практический эффект:

- Обычный collaborator с Write-ролью **вынужден** идти через PR.
- Автор-admin **технически** может push'ить прямо в `main` (bypass
  срабатывает), но по договорённости работает только через
  `develop → PR → main`. Защита от опечатки — дисциплиной, не
  enforcement.

**Четыре workflow-ручки** (разделение по стабильности процесса):

- **`sync-wiki.yml`** — редко, через **Pull Request**. Парсинг вики
  нестабилен (изменение вёрстки → парсер может вернуть 40 мехов вместо
  97), PR-diff — обязательный review до merge. Sanity-check: сверка
  числа мехов с публичным CSV калькулятора
  `new.mechs.su/cms/wp-content/uploads/mechsearth/calc/mechs.txt`
  (`wc -l`). Сломано → PR закрывается без merge.
- **`sync-sheets.yml`** — часто, **прямой push в `main`**. Google Sheets
  API стабилен, overlay-правки точечные, редакторам не ждать review.
- **`merge-develop-to-main.yml`** — merge коммитов `develop → main`
  через `git merge --no-ff`. Триггер — только `workflow_dispatch`
  (ручная кнопка), автомерджа по крону нет (снят осознанно — merge в
  main запускается вручную). Early-exit, если в `develop` нет новых
  коммитов (не создаёт пустой merge-commit, не триггерит deploy).
  Конфликт → job падает, разрешать локально
  (`git checkout develop && git merge origin/main`, push, снова кнопку).
- **`deploy.yml`** — **автомат по push в `main`** (path-фильтр `data/`,
  `src/`, `package.json`, config) + ручной повтор через
  `workflow_dispatch`.

**Идентичность push'а в `main`.** Sync-workflow'ы, пушащие в `main`
напрямую (`sync-sheets`, `merge-develop-to-main`), используют
fine-grained PAT из secret `MERGE_PAT` (scope `Contents: Read & write`):
checkout под ним делает push от имени автора и попадает под Write-bypass.
Дефолтный `GITHUB_TOKEN` не подходит — он работает под identity
`github-actions[bot]`, а Write-bypass покрывает только людей-collaborator'ов.

Права на `workflow_dispatch` — у Collaborators с write-доступом
(2-3 человека).
