# Roadmap

Долгосрочный план развития по MVP-эпикам. Живой документ — правится по
мере продвижения. Оперативные задачи текущего MVP — в
`context/views/active.todo`. Крупные решения по факту — в
`CHANGELOG.md`.

Порядок MVP отражает зависимости, а не жёсткую очередь: MVP2 и MVP3
допускают параллельную работу, если ресурсы позволяют.

## MVP1 · Каталог мехов (карточная база)

**Цель.** Уйти от таблицы к карточному каталогу как фундаменту для всех
последующих эпиков. Снять личную боль автора «таблица достала».

**Состав.**

- Сетка карточек с иллюстрациями и ключевыми параметрами.
- Страница/модалка одного меха с полным набором параметров.
- Избранное переносится из текущего кода — механика будет использоваться
  в MVP2 (личный склад) и MVP3 (клановый склад).
- Существующая таблица (`RobotsGrid`) и механика `base-robot-comparison`
  — на снос. Advanced-режим с таблицей не сохраняем.

**Зависимости.**

- Сквозной эпик «Данные»: парсинг `new.mechs.su/wiki/robots` — оттуда же
  берём иллюстрации мехов.

**Риски.**

- Стабильность парсера при обновлениях вёрстки вики — источник HTML, а
  не API. Часть параметров может потребовать overlay.
- Визуальный дизайн карточек — отдельная фаза, проходит через Perplexity
  и Figma до Edit'ов в коде.

## MVP2 · Калькулятор прокачки

**Цель.** Личная боль автора: «что мне нужно, чтобы прокачать меха до
уровня N — какие ресурсы, что у меня уже есть, чего не хватает».

**Состав.**

- Расширение модели меха: уровни, стоимость перехода между уровнями.
- Модель ресурсов игры.
- Инвентарь пользователя («что у меня есть») — локально, IndexedDB.
- UI: выбор меха + целевой уровень → таблица дефицита ресурсов.
- Референс UI/логики: онлайн-калькулятор `new.mechs.su/help/upgrade-robots`
  (клиентский Redux — можно вытащить параметры и модель расчёта).

**Зависимости.**

- MVP1 (каталог как точка входа для выбора меха).
- Схема данных мехов расширяется — синхронно с эпиком «Данные».

**Риски.**

- Детальная таблица стоимости прокачки может отсутствовать в вики.
  Fallback — вытянуть из state онлайн-калькулятора либо через overlay +
  screenshot-to-patch.

## MVP3 · Клановый склад

**Цель.** Командная боль: «что есть в клане, чего не хватает, кто чем
может поделиться».

**Состав.**

- Shared backend (обсудим при подходе: Supabase, Cloudflare Workers+D1
  или другой BaaS).
- Примитивная идентификация членов клана.
- Синхронизация «личных складов» из MVP2 → «клановый агрегат».
- UI: агрегированный вид склада, дефицит по общим целям клана.

**Зависимости.**

- MVP2 (модель ресурсов и «личный склад» переиспользуются).

**Риски.**

- Появление бэкенда — архитектурная развилка. Отдельная фаза `docs`
  перед стартом эпика.
- Идентификация: минимальный shared-secret vs полноценная авторизация —
  отдельное решение.

## MVP4 · Боевой калькулятор

**Цель.** Изначальная мотивация проекта: «насколько крепкий мех
получится из моего билда; кто победит в 1v1 или групповом бою; сколько
аптечек класть».

**Состав.**

- Модель оборудования и скиллов меха.
- Модель боя (1v1, групповой), метрики выживаемости.
- Сравнение мехов в бою: spec-sheet + radar overlay.
- Возврат сравнительной механики — но уже как визуализации боевых
  сборок, а не голых характеристик.

**Зависимости.**

- MVP2 (прокачанная модель меха — фундамент боевых расчётов).

## Сквозной эпик · Данные

Подкладывается под все MVP. Три слоя:

1. **Sync.** Скрипт (Node, локально или в CI) парсит
   `new.mechs.su/wiki/robots`, нормализует в `data/robots.json` и
   сохраняет ссылки/файлы иллюстраций мехов. Первичный источник.
   `MechsEarth/System` — возможный вторичный источник для перекрёстной
   проверки, не первичный.
2. **Overlay.** `data/overrides.yml` (или .json) — уточнения / патчи /
   недостающие поля. Merge при загрузке, overlay побеждает.
3. **Screenshot-to-patch.** Отдельный CLI: скармливаешь скриншот из
   игры, VLM возвращает JSON-патч в overlay. Полуручной процесс,
   убирает боль ручного ввода.

Версионирование: `game_patch_version` в snapshot данных + git-история
overlay. Схема данных — через `schema_version`.

## Сквозной эпик · Инфраструктура

Появляется по мере надобности, не спекулятивно:

- Хостинг MVP1: GitHub Pages / Vercel — статика хватает.
- Хостинг MVP3+: платформа с поддержкой бэкенда (следствие выбранного
  BaaS).
- Домен — если/когда решим публиковать.
- Мобильная адаптация — критично для карточного UI, входит в MVP1.
