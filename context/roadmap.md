# Roadmap

Долгосрочный план развития по MVP-эпикам. Живой документ — правится по
мере продвижения. Оперативные задачи текущего MVP — в
`context/views/active.todo`. Крупные решения по факту — в
`CHANGELOG.md`.

Порядок MVP отражает зависимости, а не жёсткую очередь: MVP2 и MVP3
допускают параллельную работу, если ресурсы позволяют.

## MVP1 · Справочный слой

**Цель.** Полный справочный слой приложения: каталоги всех игровых
сущностей, автоматически обновляемые из вики + overlay для того, что
парсер не даёт или что содержит неточности. Фундамент для всех
последующих MVP.

**Data-сущности (11).** Все — в `docs/entities/data--*.yml`:

- `data--robots-catalog` ✓ — мехи (Боец, Транспортник, Добытчик, Разведчик).
- `data--weapons-catalog` — Вооружение (wiki 3341).
- `data--equipment-catalog` — Дополнительное оборудование (3682) + Чипы (3657) + Энергоснабжение (3106) + Защита (3108).
- `data--ammo-catalog` — Боезапасы (3661).
- `data--items-catalog` — Используемые предметы (3111).
- `data--loot-catalog` — Лут монстров/кротов/пиратов (3023/3027/20118). **Гибрид:** парсер + ручной ввод (вес, базовая цена без навыка «Торговля»).
- `data--blueprints-catalog` — Чертежи (3034).
- `data--ore-catalog` — Полезное ископаемое (3036).
- `data--components-catalog` — Компоненты (3955). **Гибрид:** парсер (список) + ручной ввод (рецепты крафта: чертёж + входные лут/компоненты).
- `data--currencies` — Боны (3017) + Реглы (3020).
- `data--skills-catalog` — Навыки (2902) + Умения (2905). В MVP1 — только факты (название, описание); модель модификаторов (например, «Торговля» → цена продажи) — фаза `docs` перед MVP2.

**Иконки предметов.** У каждой сущности каталога — поле `iconPath`
(относительный путь `data/icons/<catalog>/<key>.<ext>` в основном репо).
Иконки заводятся через парсер (если получается вытащить из вики) или
через скриншоты автора.

**Подход C (пилот-цепочка).** Сначала полностью автоматизируем сквозную
цепочку `wiki → parser--wiki → data/ в репе → build → view` только на
мехах. Убедились — накатываем остальные 10 каталогов через тот же
конвейер.

**UI.** MVP1 использует `view--robots-catalog` ✓ как эталон. Остальные
каталоги подключаются через переиспользуемые примитивы
`other--catalog-shell` (`CatalogHeader`, `CatalogSection`, `CatalogGrid`,
`FavoritesDnDSection`, `FilterPanel`, `SearchField` + hook'и).
Обобщение сделано в фазе refactor 2026-07-12 — по решению юзера
пораньше, до подключения второго каталога, вместе с переходом на
роутинг. Формально это преждевременное обобщение, но приняли риск
осознанно: второй каталог покажет, что переопределять / расширять.

**Навигация.** С 2026-07-12 приложение многостраничное на
react-router-dom@7 (BrowserRouter). Общая оболочка
`layout--app-shell` (AppHeader + breadcrumbs) — на всех страницах.
Маршруты MVP1:

- `/` — `view--home` (плитки разделов; пока только «Каталоги»).
- `/catalogs` — `view--catalogs-hub` (плитки каталогов; пока только «Мехи»).
- `/catalogs/mechs-catalog` — `view--robots-catalog`.
  Новые MVP (Калькулятор прокачки, Клановый склад, Боевой калькулятор)
  подключаются как новые плитки на `/` и свои поддеревья маршрутов.
  SPA-fallback на GitHub Pages — `404.html` = копия `index.html`.

**UI подсветка overlay-полей.** Значения, пришедшие поверх автопарса из
Google Sheets, помечаются в карточке/детализации визуально — тонкий
маркер (значок или подчёркивание) + tooltip с датой последнего
обновления. Даёт пользователю понимание «что тут уточнено сообществом,
а что как в вики». Использует meta-поля из merged JSON (см. эпик
«Данные»).

**Зависимости.**

- Сквозной эпик «Данные» подключён полностью (Google Sheets +
  `parser--wiki` + `parser--google-sheets` + merge с meta-полями +
  иконки в репе).
- Настройка Google Sheets (Service Account, доступ к таблице) —
  отдельной фазой `docs` перед первым прогоном парсеров.
- Настройка GitHub Actions workflow (`sync-and-deploy` + Collaborators
  - деплой на GitHub Pages / Vercel) — отдельной фазой `docs`.

**Риски.**

- Стабильность парсера вики при обновлениях вёрстки — источник HTML, а
  не API.
- Объём ручного ввода для гибридных каталогов (loot, components,
  иконки) заранее неясен — может съесть много времени редакторов.
- Размер репо. Иконки в git — не идеально, но при ожидаемом объёме
  (10-50 MB, потенциально до 200 MB) терпимо. Пороги перехода на Git
  LFS или отдельное хранилище — «когда почувствуем».

**Дизайн.** MVP1 обходится MUI-defaults без выделенной дизайн-фазы —
подтверждено по факту feat #1. Возврат к дизайну — в MVP4 (radar +
spec-sheet), где defaults уже не хватит.

**Deploy prefix.** Публикация — на GitHub Pages (или Vercel) под
префиксом `/mechs/` (`<username>.github.io/mechs/`). Это закладка на
будущее — оставляет свободным `<username>.github.io/` для главной
страницы и другие префиксы для последующих проектов автора. Реализация:
`publicPath: '/mechs/'` в rspack config + `basename: '/mechs'` в
React Router (когда роутер появится).

**Отложено вне MVP1.**

- **Трафареты** (косметика, вики 3659) — не приоритет.
- **Модификация** (Установка навыков 3950, Модификация оборудования
  3940, Временный апгрейд 4019) — важные механики, но не отдельные
  каталоги. Проработка — при подходе к MVP2 (навыки) и MVP4
  (модификация оборудования, временный апгрейд).
- **Модель модификаторов навыков** (например, «Торговля» → множитель
  цены продажи лута) — отдельная фаза `docs` перед MVP2.
- **Контекстные вариации** — цены прокачки различаются между заводами
  (нейтралка v2 vs v3), лут различается между локациями, боевые
  характеристики зависят от карты (v2 vs v3). Модель ещё сырая —
  накапливаем факты через разведку в игре, эволюционируем схему при
  подходе к MVP2 (или MVP4 для боевых карт). Возможные направления:
  nested `contexts` внутри записи overlay, или отдельные каталоги вроде
  `data--upgrade-costs-by-factory`.
- **Полноценный попап overlay-подсветки** с ссылкой на источник (строка
  в Sheets) — MVP2 или отдельное улучшение UX. MVP1 обходится
  значок + tooltip с датой.
- **Форма ввода overlay** — идея на MVP3+, если объём ручных данных
  вырастет и Sheets станет неудобен.
- **Migration на Git LFS или отдельное хранилище** — когда размер
  иконок начнёт мешать `git clone`. По грубой прикидке — сотни MB.

## MVP2 · Калькулятор прокачки

**Цель.** Личная боль автора: «что мне нужно, чтобы прокачать меха до
уровня N — какие ресурсы, что у меня уже есть, чего не хватает».

**Состав.**

- Расширение модели меха: уровни, стоимость перехода между уровнями.
  Формула для реглов уже известна из community-калькулятора:
  `reglsCostMultiplier = mech.level / K`, где K зависит от типа
  (FIGHTER 100 / SCOUT 60 / TRADER 90 / MINER 35).
- Модель модификаторов навыков (проработка перед стартом эпика).
- Модель контекстных вариаций стоимости прокачки (заводы v2/v3) — если
  к моменту MVP2 накопилась разведка.
- Инвентарь пользователя («что у меня есть») — локально, IndexedDB.
- UI: выбор меха + целевой уровень → таблица дефицита ресурсов.
- Референс UI/логики: онлайн-калькулятор `new.mechs.su/help/upgrade-robots`
  (React SPA от community `mechsearth`, встроен через iframe в WP —
  используется как сверка результатов и вторичный источник данных).

**Зависимости.**

- MVP1 (полный справочный слой — мехи, currencies, ore, components,
  blueprints, skills).

**Риски.**

- Детальная таблица стоимости прокачки может отсутствовать в вики.
  Fallback — вытянуть из state онлайн-калькулятора либо через overlay.
- Модель составных рецептов компонентов (что из чего крафтится) —
  ручной ввод, объём заранее неясен.

## MVP3 · Клановый склад

**Цель.** Командная боль: «что есть в клане, чего не хватает, кто чем
может поделиться».

**Состав.**

- Shared backend (обсудим при подходе: Supabase, Cloudflare Workers+D1
  или другой BaaS).
- Примитивная идентификация членов клана.
- Синхронизация «личных складов» из MVP2 → «клановый агрегат».
- UI: агрегированный вид склада, дефицит по общим целям клана.

**Зависимости.**

- MVP2 (модель ресурсов и «личный склад» переиспользуются).

**Риски.**

- Появление бэкенда — архитектурная развилка. Отдельная фаза `docs`
  перед стартом эпика.
- Идентификация: минимальный shared-secret vs полноценная авторизация —
  отдельное решение.

## MVP4 · Боевой калькулятор

**Цель.** Изначальная мотивация проекта: «насколько крепкий мех
получится из моего билда; кто победит в 1v1 или групповом бою; сколько
аптечек класть».

**Состав.**

- Модель оборудования и скиллов меха, включая механики «Модификация»
  из вики (Установка навыков, Модификация оборудования, Временный
  апгрейд).
- Модель боя (1v1, групповой), метрики выживаемости.
- Контекстные вариации боевых характеристик по карте (v2/v3) — если к
  моменту MVP4 накопилась разведка.
- Сравнение мехов в бою: spec-sheet + radar overlay.
- Возврат сравнительной механики — но уже как визуализации боевых
  сборок, а не голых характеристик.

**Зависимости.**

- MVP2 (прокачанная модель меха — фундамент боевых расчётов).
- Полный справочный слой MVP1 (weapons, equipment, ammo, items, skills).

## Сквозной эпик · Данные

Подкладывается под все MVP. Три слоя:

1. **Sync.** Универсальный `parser--wiki` (долгоживущий js/ts-скрипт)
   с конфигом «каталог → page_id-ы → правила селекторов» парсит
   подстраницы `new.mechs.su/?page_id=…`, нормализует в JSON-файлы и
   выгружает **прямо в основной репо** — `data/<catalog>.json` +
   `data/icons/<catalog>/<key>.<ext>`. Скрипт коммитит изменения от
   имени sync-бота. `MechsEarth/System` — устаревший markdown-редактор
   для игрового клиента, не программный источник, исключён.

2. **Overlay.** Источник правды — **Google Sheets** (одна таблица, лист
   на каталог). Редакторы (доверенные игроки: 2 разработчика + игроки)
   вносят уточнения непосредственно в Sheets — знакомая среда, простые
   permissions через Google Drive share. `parser--google-sheets` тянет
   данные через API (Service Account key вне репо) и генерирует
   `data/overrides/<catalog>.yml` в репе — это авто-кэш, не
   редактируется вручную. Overrides доступны для ВСЕХ каталогов, не
   только для гибридных loot/components: любые неточности автопарса
   поправляются точечно через Sheets. Формат — плоский словарь
   `<key>: { <field>: <value> }`. См. convention `data-overlay`.

3. **Merge и meta-поля.** Build-time скрипт (`scripts/build-data.ts`)
   мержит `data/<catalog>.json` (parsed) с `data/overrides/<catalog>.yml`
   и сохраняет метаинформацию по каждой сущности: какие поля пришли из
   overlay (`_meta.overlayFields`), когда обновились
   (`_meta.overlayUpdatedAt`), ссылку на источник (`_meta.overlaySource`,
   строка в Sheets). UI использует эти meta-поля для визуальной
   подсветки уточнённых данных (см. MVP1, пункт «UI подсветка
   overlay-полей»).

**Всё в основном репо (вариант α).** Без внешнего хранилища — ни S3,
ни Synology, ни R2. Причины: наш объём данных (10-50 MB, потенциально
до 200 MB) без проблем помещается в git; PR-diff при обновлении
парсенных данных даёт бесплатный review; локальная разработка = `git
clone` без ключей. Пороги перехода на Git LFS или отдельное хранилище
— «когда почувствуем боль».

**GitHub Actions как исполнитель — три отдельные ручки.**

Разделение по стабильности процесса:

- **`sync-wiki.yml`** — редкий запуск, через **Pull Request**. Парсинг
  вики нестабилен (изменения вёрстки → парсер может вернуть 40 мехов
  вместо 97 или сломанные stats). PR-diff — обязательный review до
  merge. Если сломано — PR закрывается без merge.
- **`sync-sheets.yml`** — частый запуск, **прямой push в `main`**.
  Google Sheets API стабилен, overlay-правки точечные, редакторам не
  надо ждать review.
- **`deploy.yml`** — **автомат по push в `main`** (с фильтром путей:
  `data/`, `src/`, `package.json`, config), плюс возможность повторить
  вручную через `workflow_dispatch`.

Триггер всех трёх — только вручную (кроме автоматического `deploy`),
никакого cron. Права на `workflow_dispatch` — у Collaborators с
write-доступом (2-3 человека). См. `docs/manuals/setup-github-workflow.md`.

**Сверка с калькулятором.** У калькулятора прокачки на сайте есть
публичный CSV `https://new.mechs.su/cms/wp-content/uploads/mechsearth/calc/mechs.txt`
— полный список мехов игры (~100 строк, разделитель `;`). Используется
как источник cross-check при review `sync-wiki` PR: если наш JSON
существенно расходится с calc, парсер что-то потерял. Sanity-check в
теле PR приводит `wc -l` calc-файла — сравнивать по числу мехов и по
именам легко.

**Backup Google Sheets.**

- **Встроенный** — Sheets ведёт полную revision history, откат из UI без
  настройки.
- **Неявный** — каждый прогон `parser--google-sheets` пишет snapshot в
  `data/overrides/*.yml` и попадает в git-history репозитория.
- **Программный (опционально)** — если наберётся много редакторов и
  риск ошибок вырастет, добавим GitHub Action для регулярного архивного
  дампа Sheets в `backups/gsheets/YYYY-MM-DD.json`. Пока не заводим.

**Иконки и ручной ввод в чате.** Иконки: автор (или редактор) присылает
скриншот игрового экрана — агент обрезает и складывает в
`data/icons/<catalog>/<key>.png`, поле `iconPath` в overrides указывает
на путь. Помимо Sheets, автор может присылать данные в свободной форме
(текст, скриншоты, JSON-фрагменты) — агент раскладывает по
соответствующим листам Sheets.

**Версионирование.** `game_patch_version` в snapshot данных +
git-history репозитория + встроенная revision history Sheets.

## Сквозной эпик · Инфраструктура

Появляется по мере надобности, не спекулятивно:

- **Хостинг MVP1-2:** GitHub Pages (публичный репо) или Vercel
  (свободно для приватного). Статики хватает.
- **Deploy prefix `/mechs/`** — публикуемся не в корне, а под
  префиксом. Закладка на будущее: несколько проектов автора живут под
  одним доменом. См. `publicPath` в rspack config и `basename` в
  React Router (когда появится).
- **Хостинг MVP3+:** платформа с поддержкой бэкенда (следствие
  выбранного BaaS).
- **Домен** — если/когда решим публиковать за пределы клана.
- **Мобильная адаптация** — критично для карточного UI, входит в MVP1.
